Objective of this document


This document is intended to keep a record of the file formats created by the Tango Mapper utility application.

This is a dynamic document, and as the data formats will likely change over time. For now, we suggest to communicate meta data by simple text file, comma-separated, one entity per line.

In the future, we might move to more portable, extensible formats, such as protocol buffers binary or text).

The benefit of starting the simple text files is that is the lowest common denominator for all data ingestion platforms (C++, Matlab, etc). It makes experimenting and visualizing the data easier off the bat.

Creating other data formats form there is trivial and may be considered as next steps.


This document will not provide details about the architecture of any specific Project Tango device, but is intended to be generic enough to apply to any (or most) hardware versions. Over time, some of those concepts may become deprecated, or new one may be required, but the core descriptions should still apply.


Using Tango Mapper


  • Running the "Report Failure" app mode will create a dataset on the device, including pose data, in the one of the paths:
    • /data/data/com.motorola.atap.tangomapper/files
    • /data/data/com.google.tango.tangomapper/files
  • Running "Play/Record" app mode will create a dataset, as above, but without the pose estimation data.
  • Running "Odometry" app mode will create a dataset as in "Report Failure" mode, but has a few additional manual startup options. 
  • Calibration parameters for the device are available as a text file in /sdcard/calibration.xml.  We recommend backing this file up since it will be destroyed in case of a full manual re-flash of the device partitions (OTA will preserve the file OK however).

Datasets


A complete dataset should be made of 8 text files and 2 directories:


files or directories

content

images.txt

List of all fisheye and narrow fov camera images collected, with timestamps and pose.

depth_images.t

List of all depth images collected, with timestamps.

fisheye_to_imu_transformation.txt

Fisheye to IMU transformation matrix.

depth_to_fisheye_transformation.txt

Depth sensor to Fisheye transformation matrix.

narrow_to_fisheye_transformation.txt

Narrow fov camera to Fisheye transformation matrix.

fisheye_intrinsics.txt

Intrinsic parameters of the Fisheye camera.

narrow_intrinsics.txt

Intrinsic parameters of the Narrow fov camera.

depth_intrinsics.txt

Intrinsic parameters of the depth sensor.

depth

Directory containing all the depth images.

superframes

Directory containing all the fisheye and narrow fov camera images.




Definition of the references frames.


There are 4 main sensors on the device: a depth sensor (labelled d), a fisheye camera (labelled f) a narrow field of view camera (labelled n) and an IMU (labelled i).

The letters d, f, n and i will be used throughout the document to refer to 3D point coordinates in the frames of references attached to the sensors (Xd, Xf, Xn, Xi).

A generic 3D point P will have coordinates  Xw, Xd, Xf, Xn and Xi in a local world reference frame, the depth camera reference frame, the fisheye camera reference frame, the narrow fov camera reference frame and the IMU reference frame respectively. To keep the notations simple, we will assume that Xw, Xd, Xf, Xn and Xi are expressed in homogeneous coordinates that is are written as 4-vectors [X;Y;Z;1].

The local world reference frame should be chosen such that its z coordinate point up in the real world.

Point coordinates and translations are in units of meters.


Timestamps


The images sensors (fisheye camera and narrow field of view camera) operate at 30Hz. However, the depth sensor is not guaranteed to operate at 30Hz. As a result al three sensors are not synchronized on the same clocks.


The fisheye and narrow fov capture images in sync. and the resulting images are timestamped on the android clock (denoted ts_android).


The depth camera is NOT synchronized with the other two cameras. Its depth records are also timestamped by the android clock. However, the depth samples are timestamped with an offset compared the true android time associated the capture instant.

So, we will denote the timestamp of a depth record as ts_android_offsetted.


The IMU sensor operates on a separate clock, known as the movidius clock.

The images collected by the fisheye and narrow fov cameras are also timestamped using this movidius clock, denoted ts_movidius.

As a result, the map between android clock (ts_android) and movidius closk (ts_movidius) is provided by the image stream. It is therefore essential that those two timestamps be reported for every image collected.

These timestamps, together with the image names can be reported together with the pose information, that is described in the next section.



Pose


The global pose of the device is defined as the rigid transformation between reference frames attached to the IMU and the world reference frame.  See here for the the device frame of reference used in data logging files.  It is uniquely described by a rigid transform that maps the coordinates Xi to Xw:


Xw = Tiw * Xi


Here, Tiw is a 4x4 rigid transformation matrix that maps Xi to Xw (IMU coordinates to world coordinates -- iw stands for i to w or IMU to World).

Such a rigid transformation is a 6DOF object represented by a rotation (SO(3) matrix) and a translation vector. There are several ways to encode rotations matrices: Rodrigues angles, quaternions, Euler angles, etc To avoid any confusion, we choose here to keep the entries of the 4x4 matrix as (over-determined) parametrization of the rigid transform:


Tiw = [r11 r12 r12 t1 ; r21 r22 r23 t2 ; r31 r32 r33 t3 ; 0 0 0 1];


where rij are the entries of a SO(3) rotation matrix and ti are the entries of the translation vector.


Given the way the pose is estimated (using SFM), it is known at every image captured by the device. As a result, for every image, the 12 entries rij and ti must be reported.


The pose information as well as the image names (fisheye + narrow fov image), and the associated android and movidius timestamps should therefore be reported in a single text file, each line associated to an image.


The image name may be the basename of the superframe used to contain both images (fisheye and narrow fov image), or any other string that allow a unique determination of the two associated fisheye image and narrow fov image. If it is preferred, the image name may be substituted by an image index, provided that any index can uniquely be associated to the two images file(s).


The android and movidius timestamps should be in units of seconds. It is my understanding that the android clock is analogous to a UTC clock, while the the movidius clock is comparable to a GPS clock. In any case, the two clocks do not share the same global offset.


The pose should be the 12 parameters rij and tj, the entries of the first 3 rows of the Tiw matrix, row major.

This results in a set of 15 separate entries, comma separated per line (per image):


image_basename,t_android,t_movidius,r11,r12,r12,t1,r21,r22,r23,t2,r31,r32,r33,t3


The image file basename should include the file extension, that could be pgm if the this is the format of choice.


Previously, the android timestamp was reported in a separate file from the rest of the pose information. But it would be a lot more convenient to have all 15 entries stored in a single text file.


This file can be named images.txt.



What about the Depth sensor timestamp?


As mentioned earlier in the timestamps section, the timestamps of the depth records is an offset version of the android clock.

To compute the more accurate timestamp of the depth records that is in agreement with the the android timestamp reported per image, one need to offset ts_android_offsetted by 175 ms:


ts_android = ts_android_offsetted + 0.175


This operation lets us hallucinate a per-depth record timestamp that is directly comparable with the android timestamp reported on a per image basis, and stored in the images.txt file.


it is important that the offsett android timestamp be reported for every depth record.


This can be done following an existing file output schema, based on storing in a text file, one depth record information per line, consisting of the depth image name (the base name is sufficient here), followed by its ts_android_offsetted in seconds. The two entries should be comma-separated (to remain consistent with the choice of format made for the file images.txt):


depthimage_basename,ts_android_offsetted


This file can be named depth_images.txt.

In the future, this offsetted timestamp could be turned into the true timestamp ts_android.



Relative sensor locations


The relative location of the devices may also be encoded as a set of 3 static 4x4 matrices Tfi, Tdf and Tnf:


  • Xi = Tfi * Xf

  • Xf = Tdf * Xd

  • Xf = Tnf * Xn


Specifically,

  • Tfi encode the rigid transformation between the fisheye camera and the IMU.

  • Tdf encodes the rigid transformation between the depth sensor and the fisheye camera.

  • Tnf encodes the rigid transformation between the narrow fov camera and the fisheye camera.


This convention is identical to that one currently used in the Project Tango team, that is using the fisheye camera sensor as reference image device. This makes sense since it is the main source for information for SFM.


Those 3 transformation matrices should be stored in separate text files following the same convention as for the pose transformation matrix Tiw (the first 12 row-major entries of the 4x4 matrix, all comma-separated), each file assigned a name that is descriptive of the nature of the transformation.


  • The entries of Tfi should be stored in fisheye_to_imu_transformation.txt

  • The entries of Tdf should be stored in depth_to_fisheye_transformation.txt

  • The entries of Tnf should be stored in narrow_to_fisheye_transformation.txt



Intrinsic sensor parameters


The three image devices (depth sensor, fisheye camera and narrow fov camera) must also be calibrated intrinsically, and their intrinsic parameters must also be recorded to text files.


Since they are all calibrated using the Devernay fov camera model (see http://citeseerx.ist.psu.edu/viewdoc/download?doi=10.1.1.18.7064&rep=rep1&type=pdf), they all have 8 distinct parameters, all comma-separated:


width height,fx,fy,cx,cy,omega,max_angle


In details:

  • (width, height): Size of the image

  • (fx, fy): Focal length in unit of horizontal and vertical pixels.

  • (cx, cy): Principal point location in pixels where (0,0) denotes the center of the upper left pixel.

  • omega: Field of view parameters in radians.

  • max_angle: Maximum angle corresponding to the maximum fov of the sensor in radians.


The 8 entries should be stored as a single line in a text file, one file per sensor, all entries comma-separated. The files should be named: fisheye_intrinsics.txt, narrow_intrinsics.txt and depth_intrinsics.txt
